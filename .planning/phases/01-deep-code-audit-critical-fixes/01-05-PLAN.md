<objective>
Pin the FastAPI gateway to exactly one instance/worker in IaC (AUD-04) and document it as load-bearing for the gateway's in-memory state. This is the mitigation the locked decisions chose instead of async-lock rewrites or a distributed cache: the rate-limit store, JWKS cache, and ownership fast-path map are correct ONLY at exactly one process. The Bicep must assert plan sku.capacity = 1, any autoscale maximum = 1, and pin the gunicorn startup command to a single worker.

Purpose: Re-validation under the N>1 threat model concluded a single-instance pin closes the in-memory-state-inconsistency threats (Pitfall 2). This plan makes that decision enforceable in infrastructure and self-documenting for the operator and the Phase 5 evidence pack.
Output: Modified infra/main.bicep (gwPlan capacity, gateway startup command, load-bearing comment).
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05-01 | Tampering (rate-limit bypass) | _rate_limit_store under >1 worker | mitigate | Pin sku.capacity=1 + gunicorn --workers 1; autoscale max=1; load-bearing comment. A user cannot split requests across workers to bypass per-user limits. |
| T-01-05-02 | Spoofing / Info disclosure | _JWKS_CACHE divergence | mitigate | Single process = one JWKS cache; eliminates inconsistent key state during rotation across workers. |
| T-01-05-03 | Elevation (IDOR fast-path) | _audit_job_owners cache divergence | mitigate | Single process keeps the fast-path map coherent; the durable owner_hash check (Plan 01-06) remains the authoritative gate regardless. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No package install — IaC edit only. Legitimacy gate not triggered. |
</threat_model>
