1. Objectif de l’étape 2
L’objectif de cette étape est de définir l’architecture cible d’AC360 avant toute construction technique avancée.
Cette architecture doit permettre de répondre à quatre exigences majeures :
Utilité commerciale : l’assistant doit aider concrètement les commerciaux dans leurs dossiers clients, rendez-vous, FIC et appels d’offres.
Fiabilité métier : les réponses doivent être sourcées, contrôlées et cohérentes avec les documents et les données Artus/Fabric.
Sécurité : l’assistant doit respecter les permissions utilisateur, les données sensibles et les règles de gouvernance interne.
Évolutivité : la solution doit pouvoir démarrer simplement, puis évoluer vers des modules plus avancés sans refonte complète.
2. Principe directeur d’architecture
AC360 doit être conçu comme un orchestrateur métier, pas comme un simple chatbot.
Le rôle de chaque couche doit être clairement séparé :

4. Découpage en domaines fonctionnels
4.1 Domaine 1 — Interaction commerciale
Ce domaine couvre l’usage quotidien de l’assistant par les commerciaux.
Fonctions :
poser une question sur un client ;
demander une synthèse de dossier ;
préparer un rendez-vous ;
demander les documents disponibles ;
générer un brouillon de mail ;
demander les points d’attention ;
demander une FIC brouillon ;
demander une analyse d’appel d’offres.
Canal principal : Microsoft Teams.
4.2 Domaine 2 — Recherche documentaire SharePoint
Ce domaine couvre l’accès aux documents clients.
Sources :
contrats ;
tableaux de garantie ;
synthèses des modalités ;
propositions commerciales ;
FIC ;
comptes rendus ;
pièces d’appel d’offres ;
documents juridiques et techniques accessibles.
Exigences :
respecter les permissions SharePoint ;
citer les documents utilisés ;
éviter le mélange entre clients ;
distinguer les documents actifs, obsolètes, archivés ou douteux ;
demander une clarification si le client est ambigu.
4.3 Domaine 3 — Intelligence documentaire Azure
Ce domaine couvre l’OCR et l’extraction automatique.
Outil principal: Azure AI Document Intelligence.
Rôle :
lire les PDF scannés ou semi-structurés ;
extraire le texte ;
extraire les tableaux ;
détecter les champs clés ;
identifier les dates, montants, numéros de contrat, garanties, modalités ;
préremplir les métadonnées ;
alimenter des tables exploitables dans Fabric.
Objectif : réduire le metadata management manuel, sans supprimer la validation humaine sur les champs critiques.
4.4 Domaine 4 — Données Artus dans Fabric
Ce domaine couvre l’exposition des données Artus.
Fabric doit devenir la couche de vérité analytique et de contrôle pour les données issues d’Artus.
Rôle :
centraliser les données Artus utiles ;
normaliser les clients, contrats, garanties, paiements et modalités ;
exposer des tables propres ;
calculer les rapprochements ;
fournir à AC360 des résultats fiables et explicables.
Principe :
AC360 ne doit pas interroger Artus de manière brute ou non contrôlée. Les données doivent passer par une couche Fabric validée.
4.5 Domaine 5 — Contrôles métier
Ce domaine couvre les règles de comparaison et de qualité.
Contrôles principaux :
document présent ou manquant ;
document obsolète ;
garantie présente dans le document mais absente d’Artus ;
différence de plafond ;
différence de franchise ;
différence de périodicité de paiement ;
différence de date d’effet ;
différence de catégorie assurée ;
données insuffisantes pour conclure.
Ces contrôles doivent être calculés par des règles explicites dans Fabric ou dans une couche technique contrôlée.
AC360 doit expliquer les résultats, pas inventer le contrôle.
4.6 Domaine 6 — Notifications et workflow
Ce domaine couvre les alertes et les actions contrôlées.
Outil recommandé : Power Automate, intégré à Teams et SharePoint.
Usages :
notifier un commercial quand un dossier est incomplet ;
notifier un responsable en cas d’écart important ;
demander une validation de FIC brouillon ;
signaler un point à valider par la gestion ;
signaler un point à valider par le juridique.
Principe :
Les notifications doivent être utiles, limitées et actionnables. Elles ne doivent pas devenir du bruit.

5.7 Module — Recherche juridique documentaire
Objectif
Aider à retrouver, résumer et comprendre des documents juridiques ou techniques accessibles.
Entrées

Sorties

Limite
AC360 ne donne pas d’avis juridique définitif.
Phrase obligatoire lorsque le sujet est juridique :
Cette réponse est une aide à la recherche documentaire. Elle ne remplace pas une validation juridique.

Objectif
Accélérer la préparation des réponses aux appels d’offres.
Entrées

pièces manquantes ;
points de risque ;
brouillon de réponse ;
questions à clarifier avec le chargé client.
Limite
AC360 ne produit pas une réponse finale envoyable sans validation humaine.

7. Flux métier principaux
7.1 Flux — Synthèse dossier client
Utilisateur Teams→ AC360 reçoit la demande→ Identification du client→ Recherche dans SharePoint avec droits utilisateur→ Sélection des documents pertinents→ Synthèse sourcée→ Réponse structurée dans Teams
Point de contrôle : si le client est ambigu, AC360 demande une clarification.
7.2 Flux — Complétude dossier
Utilisateur demande un contrôle dossier→ AC360 identifie le client→ Inventaire SharePoint des documents accessibles→ Classification par type documentaire→ Comparaison avec checklist officielle→ Liste documents présents / manquants / douteux→ Réponse Teams→ Notification éventuelle au commercial responsable
Point de contrôle : la checklist doit être validée par le métier.
7.3 Flux — OCR et préremplissage documentaire
Nouveau document déposé dans SharePoint→ Déclenchement pipeline contrôlé→ Azure AI Document Intelligence lit le document→ Extraction texte, tableaux, champs clés→ Stockage résultat d’extraction→ Préremplissage métadonnées proposées→ Validation humaine si champ critique→ Données exploitables par AC360 et Fabric
Point de contrôle : les champs critiques ne doivent pas être considérés comme officiels sans validation ou règle de confiance acceptée.
7.4 Flux — Rapprochement document vs Artus/Fabric
Document structuré via OCR ou métadonnées→ Données Artus disponibles dans Fabric→ Matching client / contrat→ Règles de comparaison Fabric→ Résultat dans fact_controle_guide_gestion→ AC360 restitue les écarts→ Action recommandée commercial / gestion / juridique
Point de contrôle : AC360 explique les écarts calculés, mais ne les invente pas.
7.5 Flux — Brouillon FIC
Utilisateur demande une FIC brouillon→ AC360 identifie le client→ Récupère documents utiles→ Récupère données Fabric validées si disponibles→ Récupère modèle FIC→ Génère brouillon structuré→ Liste champs incertains et sources→ Demande validation humaine
Point de contrôle : pas de mise à jour automatique du document officiel en première version.
7.6 Flux — Assistance appel d’offres
Dossier AO disponible dans SharePoint→ AC360 analyse les documents accessibles→ Extrait exigences principales→ Compare avec pièces disponibles→ Identifie documents manquants→ Prépare brouillon de réponse→ Liste validations nécessaires
Point de contrôle : la réponse finale reste sous responsabilité humaine.
8. Sécurité et gouvernance
8.1 Principes de sécurité
Authentification Microsoft/Entra obligatoire.
Pas d’accès anonyme.
Respect strict des permissions SharePoint.
Respect des droits Fabric.
Pas de compte technique surpuissant exposant toutes les données.
Pas de recherche Web.
Pas de source personnelle par défaut.
Pas d’écriture documentaire officielle en première version.
Journalisation des usages sensibles.
Refus explicite en cas de demande hors périmètre.

8.3 Gestion des droits
Principe utilisateur
L’utilisateur doit voir uniquement ce qu’il peut déjà voir dans SharePoint ou dans Fabric.
Principe outil
Les outils connectés à Fabric ou SharePoint doivent éviter de contourner les droits utilisateur.
Principe audit
Les accès et les réponses doivent être suffisamment traçables pour analyser un incident ou une réponse contestée.

1. Résumé exécutif
AC360 est un assistant commercial interne destiné à aider les équipes commerciales, les responsables client, la gestion et les fonctions support à exploiter plus efficacement les dossiers clients, tout en respectant strictement la sécurité, les droits d’accès et la gouvernance documentaire de l’entreprise.
L’objectif n’est pas de créer un simple chatbot documentaire. L’objectif est de construire progressivement un produit entreprise capable de :
retrouver et synthétiser les documents clients ;
préparer les rendez-vous commerciaux ;
vérifier la complétude des dossiers clients ;
assister la production des FIC, fiches d’information commerciales ;
comparer les informations documentaires avec les données de gestion issues d’Artus via Microsoft Fabric ;
exploiter les documents juridiques et techniques accessibles ;
assister les réponses aux appels d’offres ;
notifier les commerciaux en cas de document manquant ou de point d’attention ;
fournir des réponses sourcées, traçables et contrôlées.
AC360 doit être conçu comme un produit sécurisé, gouverné et mesurable. Il doit apporter un gain commercial réel tout en limitant les risques d’hallucination, de fuite d’information, de mauvaise interprétation juridique ou d’erreur métier.
2. Vision produit
2.1 Vision cible
AC360 doit devenir un copilote commercial et métier sécurisé, intégré à Microsoft Teams, SharePoint, Microsoft Fabric et aux outils Azure d’intelligence documentaire.
Sa valeur principale est de transformer les dossiers clients dispersés en informations exploitables, contrôlées et actionnables par les équipes commerciales.
2.2 Positionnement recommandé
AC360 est un assistant commercial sécurisé connecté aux dossiers clients SharePoint et aux données de gestion Artus/Fabric. Il aide les équipes commerciales à préparer leurs rendez-vous, contrôler la complétude des dossiers, produire des brouillons de FIC, identifier les incohérences entre documents et paramétrage réel, exploiter les contenus juridiques/techniques et accélérer les réponses aux appels d’offres.
2.3 Ce que AC360 n’est pas
AC360 n’est pas :
un chatbot généraliste ;
un moteur de recherche libre sur tout Microsoft 365 ;
un outil de modification automatique des documents officiels ;
un outil de décision juridique ;
un substitut au commercial, à la gestion ou au juridique ;
un système qui contourne les permissions SharePoint ou Fabric ;
un agent autonome non contrôlé.

8. Règles de sécurité et de confidentialité
AC360 doit respecter les principes suivants :
L’utilisateur ne voit que les informations auxquelles il a déjà accès.
L’agent ne doit jamais révéler l’existence ou le contenu d’un document non accessible.
L’agent ne doit jamais mélanger deux clients.
Si le client est ambigu, l’agent doit demander une clarification.
L’agent doit citer les sources utilisées lorsque c’est possible.
L’agent doit distinguer clairement les faits, les incertitudes et les recommandations.
Les actions sensibles doivent rester soumises à validation humaine.
Aucune écriture dans les documents officiels ne doit être réalisée dans la première version.
Message de fallback recommandé :
Je n’ai pas trouvé d’information accessible sur ce client dans les documents disponibles.

Assistant Client 360Agent Copilot connecté aux dossiers clients SharePoint
Note de cadrage pour validation d’un POCObjectif : permettre aux équipes commerciales de dialoguer avec un agent IA à propos d’un client, d’un document ou d’un détail contractuel, directement depuis l’environnement Microsoft 365.
Résumé en une phrase
Créer un assistant commercial sécurisé, branché sur les dossiers clients SharePoint, capable de rechercher, résumer et citer les documents pertinents afin de préparer plus vite les rendez-vous, réduire les allers-retours internes et fiabiliser la connaissance client.
Document préparé le 28/04/2026
1. Synthèse exécutive
Les équipes commerciales disposent aujourd’hui de dossiers clients dans SharePoint : contrats, propositions, documents administratifs, comptes rendus, avenants, échanges et pièces de suivi. La valeur de ces documents est forte, mais leur exploitation reste souvent manuelle : recherche de fichiers, lecture de documents longs, vérification de versions et préparation de synthèses avant rendez-vous.
La proposition est de mettre en place un agent “Assistant Client 360” accessible depuis Microsoft 365 Copilot, Teams ou SharePoint, capable de répondre à des questions en langage naturel à partir des documents SharePoint autorisés. L’agent s’appuie sur une logique RAG : il recherche les documents pertinents, récupère les passages utiles, génère une réponse synthétique et renvoie les sources consultées.
Décision demandée
Valider un POC limité à 10 à 20 dossiers clients représentatifs, sur une durée courte, afin de tester la valeur métier, la qualité des réponses, la sécurité d’accès et l’adoption par les commerciaux.
2. Problème métier à résoudre
Constat
Impact

Les commerciaux perdent du temps à retrouver le bon document ou la bonne version dans les dossiers clients.
Connaissance client dispersée
Les informations utiles sont réparties entre contrats, propositions, comptes rendus et documents administratifs.
Risque d’erreur ou d’oubli
Un détail contractuel, une date d’échéance ou une clause importante peut être manqué avant un rendez-vous.
Préparation commerciale non standardisée
Les synthèses avant rendez-vous dépendent du temps disponible et de la connaissance individuelle du dossier.
Onboarding plus long
Un nouveau collaborateur doit parcourir beaucoup de documents avant d’avoir une vision fiable d’un client.
3. Solution proposée
Mettre en place un agent conversationnel spécialisé sur les dossiers clients. L’utilisateur pose une question simple, par exemple “Résume-moi le dossier du client X avant mon rendez-vous” ou “Retrouve la clause de résiliation du contrat Y”. L’agent interroge les contenus SharePoint auxquels l’utilisateur a accès, produit une réponse structurée et cite les documents sources.

1. Objectif de l’étape 5
Cette étape marque le passage du cadrage fonctionnel vers la conception technique concrète.
L’objectif est de définir précisément les composants à créer, leur rôle, leurs interactions et les règles techniques à respecter pour construire AC360 proprement.
Le but n’est pas encore de tout développer. Le but est de figer le socle technique qui permettra ensuite d’implémenter sans se disperser.
2. Principe technique général
AC360 doit être construit comme une architecture Microsoft intégrée :
Microsoft Teams  ↓Copilot Studio — Agent AC360  ↓Actions / rubriques / outils contrôlés  ↓SharePoint — documents clientsAzure AI Document Intelligence — OCR/extractionMicrosoft Fabric — Artus, contrôles, rapprochementsPower Automate — workflows et notificationsAzure — fonctions techniques, Key Vault, logs, sécuritéEntra ID — identité, groupes, permissions
Règle technique principale :
Copilot Studio orchestre la conversation, mais les traitements critiques sont externalisés dans des composants contrôlés : Fabric pour les contrôles métier, Azure Document Intelligence pour l’extraction documentaire, Power Automate ou Azure Functions pour les flux techniques.

5. Identité et sécurité d’exécution
5.2 Principe d’accès utilisateur
AC360 doit respecter les droits utilisateur.
Pour SharePoint :
l’utilisateur voit uniquement les documents auxquels il a accès ;
les bibliothèques sensibles ont des permissions dédiées ;
les dossiers clients confidentiels ne doivent pas être visibles par défaut.
Pour Fabric :
l’utilisateur ou l’outil doit accéder uniquement aux vues/tables validées ;
les données Artus brutes ne doivent pas être exposées directement ;
les tables de contrôle doivent être filtrées si nécessaire.
5.4 Identités managées
Pour les composants Azure :
Azure Functions avec managed identity ;
accès Key Vault par managed identity ;
accès Storage par managed identity ;
accès aux ressources Azure limité au strict nécessaire.
6. Flux technique 1 — Question simple dossier client
1. Utilisateur pose une question dans Teams.2. Copilot Studio reçoit la demande.3. AC360 identifie le client demandé.4. Si le client est ambigu, AC360 demande une clarification.5. AC360 recherche dans les connaissances SharePoint autorisées.6. AC360 synthétise uniquement les documents accessibles.7. AC360 cite les documents utilisés si possible.8. AC360 répond dans Teams avec la structure standard.9. Interaction journalisée de manière minimale.
Critère de réussite : l’agent ne répond jamais avec une information non accessible ou non sourcée sur un sujet sensible.
7. Flux technique 2 — Nouveau document SharePoint avec OCR
1. Un document est déposé dans SharePoint.2. Un flux Power Automate détecte l’ajout.3. Le document est placé dans une file de traitement ou traité directement.4. Azure AI Document Intelligence analyse le document.5. Les résultats OCR sont stockés dans une table technique ou fichier JSON.6. Les champs critiques sont marqués A_VALIDER.7. Les métadonnées SharePoint peuvent être proposées ou préremplies selon le niveau de confiance.8. Une tâche de validation est créée si nécessaire.9. Les résultats validés alimentent Fabric.
Critère de réussite : aucun champ critique ne devient automatiquement vérité métier sans règle de confiance ou validation définie.
8. Flux technique 3 — Contrôle de complétude dossier
1. L’utilisateur demande : “Le dossier du client X est-il complet ?”2. AC360 identifie le client.3. Un outil technique récupère l’inventaire documentaire SharePoint.4. L’outil récupère la checklist applicable.5. L’outil compare documents présents vs documents attendus.6. Le résultat est stocké ou retourné à AC360.7. AC360 explique les documents présents, manquants et douteux.8. Si le manque est important, Power Automate peut notifier le commercial responsable.
Table résultat recommandée :
fact_controle_completude- controle_id- client_id- date_controle- document_attendu- document_present- document_id_trouve- statut_controle- criticite- commentaire
Statuts possibles :
PRESENTMANQUANTOBSOLETEA_VALIDERNON_APPLICABLE
9. Flux technique 4 — Rapprochement guide de gestion vs Artus/Fabric
1. Le tableau de garantie et la synthèse des modalités sont disponibles dans SharePoint.2. Azure Document Intelligence extrait les garanties, plafonds, franchises, modalités, dates.3. Les données extraites sont validées ou qualifiées par niveau de confiance.4. Les données Artus sont disponibles dans Fabric via tables validées.5. Fabric exécute les règles de comparaison.6. Les écarts sont écrits dans fact_controle_guide_gestion.7. AC360 récupère les résultats du contrôle.8. AC360 explique les écarts au commercial ou à la gestion.
Table résultat recommandée :
fact_controle_guide_gestion- controle_id- client_id- contrat_id- champ_controle- valeur_document- valeur_artus- statut_comparaison- niveau_risque- regle_controle- source_document_id- date_controle- commentaire_ecart

10. Flux technique 5 — Brouillon FIC
1. L’utilisateur demande une FIC brouillon.2. AC360 identifie le client.3. AC360 récupère les documents client accessibles.4. AC360 récupère les données Fabric validées si disponibles.5. AC360 récupère le modèle FIC applicable.6. AC360 génère une FIC brouillon.7. La réponse liste les champs remplis, sources, incertitudes et informations manquantes.8. Aucun document officiel n’est modifié.

1. Le dossier AO est déposé dans SharePoint.2. OCR ou lecture documentaire extrait les exigences.3. AC360 identifie les pièces demandées.4. AC360 compare avec les documents disponibles.5. AC360 signale les manques.6. AC360 prépare un brouillon de réponse.7. Validation humaine obligatoire avant envoi.
